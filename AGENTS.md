# 仓库约定

- 除非用户明确要求使用其他分支或发起 Pull Request，完成变更后默认直接提交并推送到 `main` 分支，不创建功能分支。
- Git 远端：`git@github.com:giraffe-tree/awesome-web-effects.git`。用户要求推送且 SSH 失败时，改用 `git push https://github.com/giraffe-tree/awesome-web-effects.git main:main`。
- `demo/data/effects.js` 只保存已经通过策展准入的发布目录；`demo/data/demo-admission.js` 保存评分门槛与评分记录；被拒候选只能保留在带日期的研究审计中。
- 每个发布效果必须有真实预览、六维评分和总分，且总分不低于 80；艺术完成度、动效编排、辨识度和证据质量还必须分别通过核心最低分。不得为了补齐分类或保留知名项目降低门槛。
- 特效是主实体。“推荐实现”和“观察到该效果的 AI 官网”是不同关系，不得混为实现归属；每种特效最多关联 3 家有证据的公司，无法验证时不添加。
- GIF 是效果证据，不是装饰缩略图。发布态 GIF 必须满足下述“真实预览门禁”；任一项不满足即不得合入或发布。
- `README.md` 与 `README.zh-CN.md` 由 `scripts/build-docs.mjs` 生成，不要手改目录表格。
- 每个效果条目都必须提供与推荐实现对应的最小代码，并在站点详情中可见、可独立复制；不得只给 GIF、项目链接或 Agent Prompt。最小代码不等于完整可运行 demo，界面与数据不得混淆二者。

## 真实预览门禁

### 1. 真实性与来源

- 每个 GIF 只能来自以下两类来源之一：
  1. 仓库内可运行、可复用的真实 demo，录制内容由该条目所推荐的实现、库或浏览器能力实际渲染；
  2. 有可验证出处且允许在本仓库使用的官方素材。
- 新预览和存量迁移默认优先在仓库内编写最小、真实、可复用的 demo，并从其浏览器运行结果捕获 GIF；只有无法合理本地复现，且官方素材的来源与使用依据均已核验时，才考虑采用官方素材。不得因为官方素材获取方便而跳过可复用 demo。
- 禁止使用抽象占位动画、随机生成动画、按分类复用的通用模板或与条目实现无关的视觉近似。
- 禁止通过角落色块、微小“指纹”、标题文字或其他不影响主体效果的差异，把同一模板伪装成不同预览。不同条目的 GIF 必须在主体行为上能被一眼区分。
- “概念重制”、效果猜测和无法验证的复刻只能作为本地研究材料，不得进入发布数据、README、站点或发布产物，也不得标记成真实预览。

### 2. Provenance（来源记录）

- 每个发布 GIF 必须有可追溯的 provenance，至少记录：条目 ID、来源类型（`demo` 或 `official`）、demo 源码路径或官方原始 URL、使用的库/版本或上游版本、生成/获取日期，以及对应 GIF 输出路径。
- demo 来源必须能从 provenance 直接定位到实际运行源码；官方素材来源必须能从公开页面或上游仓库核验，不得只记录 CDN 缓存地址、搜索结果页或无法确认作者的转载地址。
- 官方素材还必须记录许可或使用依据；来源、作者或使用权限不清楚时，不得收录。
- GIF、provenance 与目录条目必须一一对应。缺失、过期、错配或无法复现的记录等同于没有预览。

### 3. Demo source（真实 demo）

- demo 必须是最小但完整的可运行实现，包含所需 HTML/CSS/JS、依赖声明和启动方式；不得只有录屏专用的隐藏脚本或无法独立运行的片段。
- demo 必须调用条目声明的真实实现/API，并展示该条目的核心交互或动画机制。仅画出相似图形、用另一套技术模拟，或只展示静态终帧均不合格。
- demo 应可被读者复制、修改和复用；必要的交互说明、输入方式和兼容性要求必须与源码放在一起。
- demo 的默认视口和初始状态必须稳定、无随机漂移。若效果本身含随机性，必须固定种子或记录可复现输入。
- 第三方页面不得通过 iframe 或临时远程状态冒充本地 demo；依赖外部服务时必须明确记录并提供可重复的降级或固定输入。

### 4. Capture（可复现录制）

- GIF 必须由真实 demo 的浏览器运行结果录制，或由已核验的官方素材转换得到。不得在录制后替换主体画面来制造效果。
- 每个 demo 预览必须有可重复的 capture 配置或命令，至少固定：入口、视口、设备像素比、等待条件、交互步骤、录制时长、帧率和输出尺寸。
- 录制前必须等待字体、图片、模型、着色器及其他关键资源加载完成；需要交互的效果必须自动执行能代表核心体验的真实操作。
- 裁切、缩放、调色和压缩只能用于呈现与体积优化，不能改变效果含义、隐藏错误或制造源码中不存在的行为。
- 生成过程必须可在干净环境中重跑；只存在于个人浏览器缓存、登录态或手工操作记录中的结果不得作为最终产物。

### 5. 视觉验收

- 验收者必须同时查看运行中的 demo 和生成的 GIF，确认二者主体行为、时序、构图和交互结果一致。
- GIF 首个可读片段应在无需查看标题的情况下表达核心效果；不能只靠条目名称、角标或说明文字区分项目。
- 验收必须检查：画面是否加载完整、核心动作是否发生、循环是否自然、文字是否可读、是否无空白/闪屏/断帧/控制台错误，以及缩略图尺寸下是否仍能辨认效果。
- 外观相近的相邻条目必须做并排检查。若主体视觉或运动轨迹实质相同，必须重新设计真实 demo 的展示场景或暂缓发布，不能添加装饰性差异蒙混通过。
- 对“能否一眼看出这个效果”没有把握时，结论一律视为不通过。

### 6. 批量变更门禁

- 批量生成或替换前，先抽取少量代表性条目完成 demo、capture、provenance 和人工视觉验收；样本未通过前不得扩大批次。
- 批量任务必须逐条验证来源和录制结果，禁止让未命中的条目回退到随机/通用模板，也禁止静默复用其他条目的 GIF。
- 批量变更必须产出可审计报告，至少列出：成功项、失败项、跳过原因、来源类型、demo 路径/官方 URL、输出文件、重复内容检测结果和视觉验收状态。
- 对 GIF 做内容哈希和感知重复检查。完全重复或高度相似的结果默认阻断；只有同一官方素材确有充分理由服务多个条目时，才可在 provenance 中解释并接受人工复核。
- 不得用“先占位后补充”的方式发布。缺少合格真实预览时，应明确显示“暂无真实预览”，而不是生成貌似完整的假 GIF。
- 修改生成器、capture 流程或大量 GIF 后，必须先运行审计与验证，再逐项抽查变更预览；脚本通过不能替代人工视觉验收。

提交前运行：

```bash
node scripts/build-docs.mjs
node scripts/audit-preview-authenticity.mjs
node scripts/validate.mjs
git diff --check
```

修改真实 Demo 或 GIF 时，额外运行：

```bash
npm ci --prefix demo/preview-demos
npm run build --prefix demo/preview-demos
python3 scripts/capture-real-preview-gifs.py --built --skip-install
```

修改目录卡片、详情弹窗或复制交互时，额外运行：

```bash
python3 scripts/test-demo-catalog-ui.py
```
