# 从“展示特效”到“讲清体验”：三个交互 Demo 的审美优化方法

日期：2026-07-21（Asia/Shanghai）

关联提交：`f4fd76f`（`Polish three image-driven interaction demos`）

## 结论

这次优化最重要的经验不是换了一套配色、加了几张更好看的图片，而是把 Demo 从“证明代码会动”推进到了“让人立即理解这个效果为什么值得用”。

一个完成度高的交互 Demo，需要同时成立四层内容：

1. **机制真实**：名称所声称的库、算法或浏览器能力确实在运行。
2. **场景可信**：画面像一个真实产品片段，而不是动画测试台。
3. **视觉成体系**：图像、字体、颜色、比例、留白和动效服务于同一个方向。
4. **预览会叙事**：短 GIF 在有限时间内呈现初始状态、核心变化和稳定结果。

只完成第一层，Demo 可以通过技术验收，但通常不会好看；只完成后三层，则容易变成与条目名称不符的概念动画。审美优化真正困难的地方，是让四层相互证明。

## 一、选题：优先改“机制正确，但表达失真”的 Demo

不是所有视觉普通的 Demo 都值得先改。最有优化杠杆的对象通常有以下特征：

- 效果本身依赖具体媒介，例如照片、商品、地图、文章或数据，但当前只用了抽象色块。
- 技术行为已经真实，问题主要在展示层，因此可以在不改变条目含义的前提下升级。
- 缩略图里看不出使用价值，必须读标题才知道它想表达什么。
- 加入一个可信场景后，效果的输入、变化和结果能形成完整叙事。
- 一组新素材可以被多个相关 Demo 共享，但每个 Demo 仍有不同的主体行为。

本次三个条目都属于“图片是机制输入，不只是背景装饰”。因此采用一套旅行建筑影像，可以形成统一艺术方向；同时，BlurHash、取色过渡和 Hover Marquee 的视觉行为又足够不同，不会变成同一模板的三次换皮。

一个实用的选题判断句是：

> 如果去掉标题，用户能否在三秒内说出画面正在解决什么体验问题？

如果答案是否定的，而机制又确实值得保留，这个 Demo 就是优先优化对象。

## 二、先写体验句，再写 CSS

开始设计前，先给每个 Demo 写一句不包含库名和技术名的体验描述：

- BlurHash：**网速不够快时，住宿图片也不应该先出现一块毫无关系的灰色。**
- Palette：**切换目的地时，整个界面的氛围应该跟随照片一起变化。**
- Marquee：**浏览目的地菜单时，当前选项可以在原位置展开成一条富媒体预览。**

这句话有三个作用：

1. 决定需要什么内容，而不是先堆装饰。
2. 约束动效发生在哪里、由什么触发、变化什么。
3. 帮助视觉验收：如果最终 GIF 没有表达这句话，就算画面精致也需要返工。

审美优化常见的失控方式，是从渐变、阴影、圆角和字体开始。这些都是末端选择。先确定体验句，视觉细节才有判断依据。

## 三、素材不是装饰，而是交互系统的输入

### 1. 用一组素材建立艺术方向

本次生成了三张本地建筑旅行图片：

- `coastal-retreat.jpg`：石灰岩海岸居所、清晨、钴蓝色开口。
- `courtyard-pool.jpg`：鼠尾草绿色庭院泳池、橄榄树、浅色灰泥。
- `desert-observatory.jpg`：锈红色沙漠观测站、蓝调时刻、琥珀色室内光。

素材详情与生成说明见 [`demo/preview-demos/assets/aesthetic-collection/README.md`](../demo/preview-demos/assets/aesthetic-collection/README.md)。

三张图保持了共同的编辑摄影语言：横向构图、克制建筑、自然材质、安静光线；同时又用蓝、绿、橙建立足够明显的场景差异。这样既能让三个 Demo 属于同一视觉世界，又能让取色和切换动效拥有清楚的颜色跨度。

### 2. 图片生成时的约束比形容词更重要

有效的素材提示不只写“高级、电影感、好看”，还要规定它如何进入界面：

- 要求横向景观构图，适配 16:9 或接近 16:9 的卡片。
- 明确主体位置和留白区域，避免裁切后失去焦点。
- 不要人物、车辆、文字、Logo 和水印，减少版权、可读性与生成瑕疵风险。
- 同组图片使用一致的摄影语言，但场景主色要有差异。
- 控制细节密度；缩略图尺寸下，建筑轮廓和主色块仍应成立。
- 输出后统一转为项目内固定尺寸与格式，禁止依赖远程图片地址。

最终 PNG 被缩放为 960 像素宽的 JPEG，既保留了缩略图需要的细节，也控制了仓库和 GIF 的体积。更关键的是，本地素材让构建、录制和未来复现不依赖网络状态。

### 3. 同一张图在三个 Demo 中承担不同职责

- 在 BlurHash 中，照片是编码源和最终清晰结果。
- 在 Palette 中，照片是颜色算法的真实数据源。
- 在 Marquee 中，照片是导航选项被激活后的内容反馈。

共享素材不等于共享模板。判断是否“换皮”的关键不是图片相不相同，而是主体行为是否相同。

## 四、三个案例的具体经验

### 案例 A：BlurHash 必须让占位图与最终照片有因果关系

优化后的实现没有手工画一张“看起来像模糊照片”的渐变图，而是：

1. 通过浏览器 `Image` 加载本地海岸图片，并等待 `decode()` 完成。
2. 将图片采样到 48×27 的离屏 Canvas。
3. 从真实像素编码 4×3 分量的 BlurHash。
4. 再将该 Hash 解码为低分辨率占位图。
5. 在 3 秒确定性时间线上，从占位图自然过渡到原图。

这使视觉关系变得可信：模糊阶段已经包含原图的钴蓝开口、暖色石材和明暗结构，清晰阶段只是信息逐渐恢复，而不是两个无关图层做透明度切换。

视觉上加入住宿信息、价格、评分和日期，不是为了“填满画面”，而是让渐进加载拥有真实产品语境。信息层保持安静，主视觉仍然是图像从模糊到清晰的变化。

#### 踩坑：不要假设框架生命周期与版本一致

最初方案依赖 p5 的 `preload`，但当前固定的 `p5@2.3.0` 实际运行链路没有按预期执行，导致预览无法 ready。最终改为浏览器原生 `Image.decode()` 完成资源加载，再创建 p5 画布与占位图。

经验是：

- 资源加载是录制可靠性的基础，不应建立在未经当前版本验证的生命周期假设上。
- `__PREVIEW_READY__` 必须代表图片、字体和计算结果都已经准备完成，而不是只代表 DOM 已挂载。
- 视觉优化后资源更多，ready 条件要比原 Demo 更严格，而不是更宽松。

### 案例 B：图片取色要驱动界面，而不是只显示色卡

Palette Demo 对三张图片分别执行以下步骤：

1. 绘制到 64×36 的离屏 Canvas。
2. 读取每个像素的 RGB、亮度和饱和度。
3. 从暗部、全局环境色和高光区得到三组代表色。
4. 在目的地切换过程中插值三组颜色。
5. 同时驱动背景、面板、主文字、次要文字、选中状态和色卡。

总计采样 `64 × 36 × 3 = 6912` 个像素。运行断言会核对图片数量、采样总数和不同图片的校验信息，避免日后退化成写死颜色。

这里的审美关键是**建立颜色角色，而不是直接使用平均色**：

- `shadow` 提供背景深度和稳定性。
- `ambient` 承担场景识别与主要氛围。
- `highlight` 用于可读文字、微光和局部强调。

如果把平均色直接铺满全屏，结果容易灰、脏或刺眼；将采样色分配到不同界面角色，再和少量中性色混合，才能既保留照片气质，又维持信息层级。

### 案例 C：Hover Marquee 的运动必须发生在语义对象内部

第一版改造采用左侧菜单控制右侧独立图片窗口。它在视觉上已经完整，但没有准确表达“hover-activated image marquee menu”：用户看到的是常见的导航加预览面板，而不是菜单项自身变成跑马灯。

最终方案改为三条全宽导航行：

- 默认状态显示编号、目的地名称和方向提示。
- Hover、键盘 Focus 或选中某一行后，该行原内容在原地被重复的图片、名称和坐标轨道取代。
- 三行分别保留独立的 Motion 控制器，轨道方向有轻微差异。
- 状态优先级为 `hover → focus → selected`，鼠标、键盘和持久选择不会互相打架。

这个返工说明了一个重要原则：

> 视觉完成度不能补偿交互语义偏差。效果叫“行内跑马灯”，运动就应该发生在那一行里。

审美并不只是“漂亮”，还包括形式与含义之间的精确对应。越像成熟产品的画面，越容易掩盖语义已经偏离；因此需要在视觉评审时重新对照条目名称和体验句。

## 五、构图：把 Demo 当成一个产品镜头

### 1. 一张图只保留一个视觉主语

- BlurHash 的主语是“清晰度恢复”。
- Palette 的主语是“照片与环境色共同切换”。
- Marquee 的主语是“当前菜单行变成运动内容”。

标题、编号、元信息、色卡和说明都只能帮助识别主语，不能与主语争夺注意力。判断方法很简单：把画面缩小到目录卡片尺寸，如果第一眼只看到装饰文字，而不是交互变化，层级就错了。

### 2. 用少数稳定规则形成高级感

本次没有依赖复杂装饰，而是反复使用几条稳定规则：

- 大面积安静底色承载一块高信息密度照片。
- 细线和小号等宽信息用于建立编辑感，不使用过多卡片边框。
- 圆角、描边和阴影只保留一种主规格，避免组件各自发挥。
- 主标题与数据之间建立明显字号差，减少中间层级。
- 留白不是平均分配，而是围绕主视觉建立方向性。
- 强调色来自照片或场景，而不是另加一套品牌色。

“高级”通常来自选择更少、关系更清楚，而不是效果更多。

### 3. 运动也需要层级

把所有元素同时动起来，会让 Demo 看起来像动效合集。更稳妥的分层是：

1. **主运动**：条目要证明的机制，例如清晰度恢复、环境色切换、图片轨道移动。
2. **伴随运动**：帮助主运动连续，例如图片交叉淡化、背景颜色插值。
3. **静态锚点**：标题、价格、编号或边界，为观看者提供稳定参照。

每个 Demo 只保留一个主运动。伴随运动的幅度和速度都应弱于主运动。

## 六、GIF 不是录屏附件，而是三秒叙事

最终三个 GIF 都使用固定的 3 秒预览时间线，共 36 帧。录制结果为：

| Demo | 帧数 | 不同画面帧 | 文件大小 |
| --- | ---: | ---: | ---: |
| Hover Marquee | 36 | 36 | 约 150 KiB |
| Palette Transition | 36 | 36 | 约 601 KiB |
| BlurHash Reveal | 36 | 23 | 约 264 KiB |

BlurHash 的重复帧来自清晰终态的停留，这是有意的阅读节奏，不代表录制失败。对所有效果机械要求 36/36 不同帧，反而可能迫使本应停留的画面持续无意义运动。

### 一个可靠的短预览节奏

1. **0–20%：建立初始状态。** 观看者要看清“变化前是什么”。
2. **20–70%：发生核心变化。** 不要让标题动画占用最宝贵的时间。
3. **70–90%：呈现结果。** 给眼睛留下理解时间。
4. **90–100%：准备循环。** 通过停留、回落或构图连续性减少跳帧感。

录制前必须等待：

- `document.fonts.ready`
- 所有图片的 `decode()`
- Canvas、算法输出和动画控制器初始化
- Demo 自己的 runtime assertion

录制后要同时检查运行页面和 GIF。GIF 压缩可能改变渐变、细线、暗部和文字边缘；只看浏览器页面并不能代表最终目录卡片的效果。

## 七、把审美判断变成可以验收的条件

“看起来更好”太主观，不能支持批量协作。更有效的做法是把审美拆成可观察问题。

### 场景与语义

- 不看标题，能否说出这是图片加载、图片取色还是菜单预览？
- 内容是否像真实产品数据，而不是 Lorem Ipsum 式占位？
- 交互是否发生在名称声称的对象上？
- 去掉装饰后，核心机制仍然成立吗？

### 构图与层级

- 缩小到 320×180 后，主语是否仍然清晰？
- 是否只有一个主要焦点和一个主要运动？
- 文字是否承担信息，而不是充当纹理？
- 是否存在没有功能的徽章、渐变、光晕或边框？

### 颜色与素材

- 强调色是否来自内容或明确的视觉系统？
- 三个场景既统一又有足够区分吗？
- 图片裁切在所有状态下都保留主体吗？
- 素材是否本地化、可追溯、无文字和 Logo 风险？

### 动效与交互

- 核心变化是否由真实点击、Hover、Focus、键盘或时间线触发？
- 速度是否允许看清起点、过程和结果？
- 辅助动画是否抢过核心动画？
- 循环是否自然，reduced-motion 和键盘路径是否仍可用？

### 真实性与工程质量

- 宣称的算法或库是否真的参与了输出？
- 图片、字体和计算未完成时，ready 是否会保持失败？
- 缺少资源时是否明确报错，而不是显示一个看似合理的替代画面？
- Demo、构建产物、GIF 和目录 ID 是否一一对应？

## 八、多 Agent 协作的有效拆分

这次采用三个并行工作流：

1. 全量审计 148 个既有 GIF，选择最有审美提升杠杆的候选，并负责 BlurHash。
2. 审查构建与 GIF 捕获链路，并负责 Palette Transition。
3. 审查网页目录与实时预览接线，并负责 Hover Marquee。

并行化适合：候选审计、独立 Demo 实现、捕获链路检查、网页接线检查。以下环节必须集中收口：

- 统一艺术方向和素材规格。
- 检查三个 Demo 是否形成同一套模板。
- 对照条目名称做语义复核。
- 重建同一个 dist、批量录制 GIF、运行全仓库审计。
- 最终决定是否接受或返工。

Marquee 第一版被集中复核否决，就是集中收口的价值。多 Agent 提升速度，但不能让每个局部方案自行定义“完成”；所有方案需要回到同一组体验句和验收门槛。

建议每个并行任务都携带同样的交付契约：

```text
效果名称：
一句话体验：
真实机制：
使用场景：
GIF 中必须出现的三个状态：
键盘 / reduced-motion 路径：
本地素材与来源：
运行断言：
```

## 九、推荐工作流

### 阶段 1：审计和定基线

1. 浏览同类 Demo 的运行页面和 GIF，而不是只读源码。
2. 找到“真实但表达弱”的条目。
3. 保存优化前 GIF，建立视觉基线。
4. 为每个候选写一句体验描述和三个可见状态。

### 阶段 2：建立艺术方向

1. 决定场景领域，例如旅行、商品、数据工作台或内容阅读。
2. 定义字体、颜色、材质、密度和构图规则。
3. 先准备一小组可复用、本地化、有来源记录的素材。
4. 确认每个 Demo 使用同组素材时仍有不同主体行为。

### 阶段 3：实现真实机制

1. 先让算法、库与真实输入成立。
2. 再用产品场景包装机制，不改变它的语义位置。
3. 加入图片、字体和运行状态的严格 ready 条件。
4. 为关键机制添加 runtime assertion，阻止日后退化成假效果。

### 阶段 4：设计预览时间线

1. 在三秒内安排起点、变化、结果和循环。
2. 保留一个主运动，减少与机制无关的动画。
3. 使用固定时间和确定性输入，避免随机漂移。
4. 在目录缩略图尺寸下预览，而不是只看大窗口。

### 阶段 5：集中验收

1. 浏览器运行页面与生成 GIF 并排检查。

## 十一、可以复用的审美优化原则

1. **先修语义，再修样式。** 交互发生的位置比阴影参数更重要。
2. **先找场景，再找素材。** 场景决定图片需要表达什么，而不是反过来。
3. **让内容参与算法。** 图片不仅要被展示，还可以成为 BlurHash、取色、裁切或导航反馈的真实输入。
4. **一个 Demo 只有一个主语。** 所有静态信息和伴随动画都为它服务。
5. **统一不等于复制。** 可以共享艺术方向和素材，但主体行为必须一眼不同。
6. **漂亮不能替代准确。** 如果视觉方案偏离条目名称，应当返工，而不是改标题解释。
7. **把 GIF 当成最终产品。** 页面、捕获时序、压缩和缩略图识别度要一起设计。
8. **让主观判断留下证据。** 体验句、前后对比、运行断言、采样数量和视觉验收记录都能让审美协作更稳定。

最终目标不是让每个 Demo 都像一张设计作品，而是让观看者在很短时间内同时感到两件事：**这个效果是真的，以及我知道它可以用在哪里。**
