# 150 Demo 目录级审美优化规范：从“展示特效”到“讲清体验”

日期：2026-07-21（Asia/Shanghai）

关联提交：`f4fd76f`（`Polish three image-driven interaction demos`）

## 0. 文档定位与结论

本文既保留 `f4fd76f` 中三个图片驱动 Demo 的事实复盘，也把这些经验升级为适用于 150 个发布 Demo 的执行规范。它用于候选审计、方案设计、素材决策、源码实现、Capture、双尺度视觉验收、多 Agent 协作、Git 收口和批次推进，不替代仓库 `AGENTS.md` 中的真实性门禁。

文中关键词含义如下：

- **必须 / 不得**：发布门禁；不满足时不能提交或推送该 Demo。
- **应该**：默认执行；只有记录明确理由后才可偏离。
- **可以**：可选策略，不构成完成条件。
- **Demo 单元**：一个效果 ID 对应的源码、素材、构建输出、GIF、provenance、目录数据和验收记录的完整集合。
- **目录级优化**：不是让 150 个画面使用同一风格，而是让每个 Demo 在统一质量门槛下准确表达各自机制和用途。

这次优化最重要的经验不是换了一套配色、加了几张更好看的图片，而是把 Demo 从“证明代码会动”推进到了“让人立即理解这个效果为什么值得用”。

一个完成度高的交互 Demo，需要同时成立四层内容：

1. **机制真实**：名称所声称的库、算法或浏览器能力确实在运行。
2. **场景可信**：画面像一个真实产品片段，而不是动画测试台。
3. **视觉成体系**：图像、字体、颜色、比例、留白和动效服务于同一个方向。
4. **预览会叙事**：短 GIF 在有限时间内呈现初始状态、核心变化和稳定结果。

只完成第一层，Demo 可以通过技术验收，但通常不会好看；只完成后三层，则容易变成与条目名称不符的概念动画。审美优化真正困难的地方，是让四层相互证明。

## 2. “优化完成”的目录级定义

“源码改完”“新 GIF 已生成”和“看起来更漂亮”都不等于优化完成。一个 Demo 只有同时通过以下八道门，状态才能从 `in-progress` 改为 `done`：

| 门 | 完成条件 | 必须留下的证据 |
| --- | --- | --- |
| G1 机制 | 条目声明的库、算法或浏览器能力真实参与最终输出；没有用别的技术画一个近似结果 | 关键源码、runtime assertion、真实性审计结果 |
| G2 语义 | 交互发生在名称声称的对象和层级上；体验句与 GIF 可见行为一致 | 体验句、初始/过程/结果三个状态说明 |
| G3 场景 | 内容像可使用的产品片段，不依赖标题解释，也不是 Lorem Ipsum 或抽象占位 | 320×180 首帧与核心帧 |
| G4 视觉 | 只有一个视觉主语；构图、素材、字体、颜色和运动层级共同服务该主语 | 缩略图与实时 Modal 两种尺度的验收记录 |
| G5 交互与可访问性 | 真实触发路径可用；键盘、Focus、触控或 reduced-motion 按条目需要成立 | 交互路径清单和目录 UI 测试结果 |
| G6 Capture | 自动型或交互型分流正确；等待条件、输入、时序、帧率、尺寸均可复现，GIF 与运行页面一致 | Capture 配置、GIF 帧数/不同帧数/体积 |
| G7 可追溯性 | 素材、源码、GIF、manifest、provenance、目录 ID 一一对应，优化前基线可恢复 | 基线 SHA、资产来源、provenance、前后对比 |
| G8 交付 | 定向验证和全局门禁通过；该 Demo 已形成独立提交并由唯一集成人完成 push | 提交 SHA、远端包含性检查、批次台账状态 |

补充规则：

- G1、G2、G6、G7 是硬门禁，视觉再出色也不能豁免。
- 目录卡缩略图和实时 Modal 是两个独立表面；任何一个未通过，G4 都不成立。
- `done` 是远端状态，不是某个 Agent 的本地判断。尚未 push 的单元最多只能标记为 `qa-pass`。
- 一个共享素材文件可以服务多个 Demo，但每个 Demo 必须分别证明它如何参与自己的机制；共享素材不能成为合并提交的理由。

## 3. 单 Demo 逐项交付契约

每个 Demo 在开始实现前必须创建一份契约；实现者和集成人都以它为准，不在过程中私自改写效果含义。建议把契约填入批次台账的详细记录：

```yaml
effect_id: blurhash-to-photo-progressive-reveal
owner: agent-or-person
baseline:
  commit: <sha>
  gif_path: demo/gifs/captured/<effect-id>.gif
  source_paths:
    - demo/preview-demos/<effect-id>.html
    - demo/preview-demos/src/<effect-id>-demo.js
experience_sentence: "不含库名、描述用户能感知到的价值"
mechanism:
  claimed_library_or_api: "p5@2.3.0 / BlurHash"
  real_input: "本地图片像素"
  runtime_assertion: "必须证明的对象、数量或状态"
scene:
  product_context: "住宿卡 / 数据台 / 阅读器等"
  primary_visual_subject: "唯一主语"
  visible_states: [initial, core-change, stable-result]
assets:
  strategy: reuse-local | imagegen | code-native | verified-official
  files: []
  source_or_generation_record: "来源、许可或生成说明"
capture:
  type: automatic | interactive | hybrid
  viewport: 320x180@1x
  duration_seconds: 3
  fps: 12
  wait_for: [fonts, images, runtime, controller]
  action_script: "自动时间点或真实 pointer/keyboard 动作"
accessibility:
  keyboard_path: "按条目填写"
  reduced_motion_behavior: "按条目填写"
deliverables:
  source: []
  built_output: []
  gif: demo/gifs/captured/<effect-id>.gif
  provenance: demo/gifs/provenance.json
  catalog_data: []
acceptance:
  thumbnail: pending
  live_modal: pending
  authenticity: pending
  validation: pending
git:
  commit: pending
  pushed_to_main: false
```

逐项交付必须至少包含：

1. 可独立运行的 Demo HTML/JS/CSS，以及机制所需的本地素材。
2. 与条目声明一致的 ready 条件和 runtime assertion。
3. 明确的 Capture 类型、三秒叙事和真实输入步骤。
4. 重新生成且人工查看过的 GIF；不得只更新源码或只更新 GIF。
5. 必要的 manifest、provenance、目录数据和生成文档同步；不相关的全局文件不得顺手改动。
6. 可恢复的 before 基线、after 结果和双尺度验收记录。
7. 定向检查结果、全仓库门禁结果、独立提交 SHA 和 push 状态。

如果实施中发现契约里的产品场景无法准确承载机制，先把状态改为 `needs-redesign` 并重新评审体验句，不得靠标题、说明文字或装饰掩盖偏差。

## 4. 选题：优先改“机制正确，但表达失真”的 Demo

不是所有视觉普通的 Demo 都值得先改。最有优化杠杆的对象通常有以下特征：

- 效果本身依赖具体媒介，例如照片、商品、地图、文章或数据，但当前只用了抽象色块。
- 技术行为已经真实，问题主要在展示层，因此可以在不改变条目含义的前提下升级。
- 缩略图里看不出使用价值，必须读标题才知道它想表达什么。
- 加入一个可信场景后，效果的输入、变化和结果能形成完整叙事。
- 一组新素材可以被多个相关 Demo 共享，但每个 Demo 仍有不同的主体行为。

本次三个条目都属于“图片是机制输入，不只是背景装饰”。因此采用一套旅行建筑影像，可以形成统一艺术方向；同时，BlurHash、取色过渡和 Hover Marquee 的视觉行为又足够不同，不会变成同一模板的三次换皮。

一个实用的选题判断句是：

> 如果去掉标题，用户能否在三秒内说出画面正在解决什么体验问题？

如果答案是否定的，而机制又确实值得保留，这个 Demo 就是优先优化对象。

## 5. 先写体验句，再写 CSS

开始设计前，先给每个 Demo 写一句不包含库名和技术名的体验描述：

- BlurHash：**网速不够快时，住宿图片也不应该先出现一块毫无关系的灰色。**
- Palette：**切换目的地时，整个界面的氛围应该跟随照片一起变化。**
- Marquee：**浏览目的地菜单时，当前选项可以在原位置展开成一条富媒体预览。**

这句话有三个作用：

1. 决定需要什么内容，而不是先堆装饰。
2. 约束动效发生在哪里、由什么触发、变化什么。
3. 帮助视觉验收：如果最终 GIF 没有表达这句话，就算画面精致也需要返工。

审美优化常见的失控方式，是从渐变、阴影、圆角和字体开始。这些都是末端选择。先确定体验句，视觉细节才有判断依据。

## 6. 素材不是装饰，而是交互系统的输入

### 1. 用一组素材建立艺术方向

本次生成了三张本地建筑旅行图片：

- `coastal-retreat.jpg`：石灰岩海岸居所、清晨、钴蓝色开口。
- `courtyard-pool.jpg`：鼠尾草绿色庭院泳池、橄榄树、浅色灰泥。
- `desert-observatory.jpg`：锈红色沙漠观测站、蓝调时刻、琥珀色室内光。

素材详情与生成说明见 [`demo/preview-demos/assets/aesthetic-collection/README.md`](../demo/preview-demos/assets/aesthetic-collection/README.md)。

三张图保持了共同的编辑摄影语言：横向构图、克制建筑、自然材质、安静光线；同时又用蓝、绿、橙建立足够明显的场景差异。这样既能让三个 Demo 属于同一视觉世界，又能让取色和切换动效拥有清楚的颜色跨度。

### 2. 图片生成时的约束比形容词更重要

有效的素材提示不只写“高级、电影感、好看”，还要规定它如何进入界面：

- 要求横向景观构图，适配 16:9 或接近 16:9 的卡片。
- 明确主体位置和留白区域，避免裁切后失去焦点。
- 不要人物、车辆、文字、Logo 和水印，减少版权、可读性与生成瑕疵风险。
- 同组图片使用一致的摄影语言，但场景主色要有差异。
- 控制细节密度；缩略图尺寸下，建筑轮廓和主色块仍应成立。
- 输出后统一转为项目内固定尺寸与格式，禁止依赖远程图片地址。

最终 PNG 被缩放为 960 像素宽的 JPEG，既保留了缩略图需要的细节，也控制了仓库和 GIF 的体积。更关键的是，本地素材让构建、录制和未来复现不依赖网络状态。

### 3. 同一张图在三个 Demo 中承担不同职责

- 在 BlurHash 中，照片是编码源和最终清晰结果。
- 在 Palette 中，照片是颜色算法的真实数据源。
- 在 Marquee 中，照片是导航选项被激活后的内容反馈。

共享素材不等于共享模板。判断是否“换皮”的关键不是图片相不相同，而是主体行为是否相同。

### 4. 是否使用 ImageGen 的判断树

ImageGen 只解决“缺少合适的位图输入”，不能替代 CSS、SVG、数据、品牌素材或真实产品证据。每个 Demo 按以下顺序决策：

```text
机制的真实输入是否需要照片、插画或纹理？
├─ 否
│  ├─ 图形可由 CSS / SVG / Canvas / WebGL 精确表达 → 使用代码原生资产
│  └─ 只是想让空白处更热闹 → 不增加图片，先修构图与信息层级
└─ 是
   ├─ 仓库已有可追溯、许可清楚且裁切合适的素材？
   │  └─ 是 → 优先复用；记录原路径，不重复生成近似图
   └─ 否
      ├─ 画面是否必须代表真实品牌、真实人物、真实地点、新闻、数据或官方 UI？
      │  └─ 是 → 不使用 ImageGen；取得可验证的官方/授权素材，或缩小场景
      └─ 否
         ├─ 是否需要特定构图、留白、主色或同组艺术方向来证明机制？
         │  └─ 是 → 使用 ImageGen，并本地化、压缩、记录生成说明
         └─ 否 → 使用简单自有素材或代码生成，避免无必要的资产成本
```

使用 ImageGen 后仍必须通过以下检查：

- 图片是机制输入或场景证据，而不是遮掩空洞机制的背景。
- 缩小到实际目录卡片后，主体、主色和轮廓仍可辨认。
- 检查手部、建筑连接、反射、重复纹理、伪文字、Logo 和水印等生成瑕疵。
- 同组图片具有一致的摄影或插画语言，同时为交互提供足够的亮度、色相或构图差异。
- 固定为本地路径并记录生成日期、提示意图、尺寸、格式和使用 Demo；不得在运行时依赖生成服务。
- 如果后续需要表达真实商品、地点或品牌，必须更换为已验证素材，不能继续把生成图描述为事实照片。

ImageGen 的否决条件也应写进契约：当机制需要像素取色、BlurHash、裁切或视差时，图片可以生成，但算法必须读取最终提交的真实文件；不得根据提示词手工写死一组“看起来像图片”的颜色或形状。

## 7. 三个案例的具体经验

### 案例 A：BlurHash 必须让占位图与最终照片有因果关系

优化后的实现没有手工画一张“看起来像模糊照片”的渐变图，而是：

1. 通过浏览器 `Image` 加载本地海岸图片，并等待 `decode()` 完成。
2. 将图片采样到 48×27 的离屏 Canvas。
3. 从真实像素编码 4×3 分量的 BlurHash。
4. 再将该 Hash 解码为低分辨率占位图。
5. 让指针进入图片时从占位图过渡到原图，移出时回到 BlurHash；点击、触控和键盘可以锁定或复位清晰图。

这使视觉关系变得可信：模糊阶段已经包含原图的钴蓝开口、暖色石材和明暗结构，清晰阶段只是信息逐渐恢复，而不是两个无关图层做透明度切换。

视觉上加入住宿信息、价格、评分和日期，不是为了“填满画面”，而是让渐进加载拥有真实产品语境。信息层保持安静，主视觉仍然是图像从模糊到清晰的变化。

#### 踩坑：不要假设框架生命周期与版本一致

最初方案依赖 p5 的 `preload`，但当前固定的 `p5@2.3.0` 实际运行链路没有按预期执行，导致预览无法 ready。最终改为浏览器原生 `Image.decode()` 完成资源加载，再创建 p5 画布与占位图。

经验是：

- 资源加载是录制可靠性的基础，不应建立在未经当前版本验证的生命周期假设上。
- `__PREVIEW_READY__` 必须代表图片、字体和计算结果都已经准备完成，而不是只代表 DOM 已挂载。
- 视觉优化后资源更多，ready 条件要比原 Demo 更严格，而不是更宽松。

### 案例 B：图片取色要驱动界面，而不是只显示色卡

Palette Demo 对三张图片分别执行以下步骤：

1. 绘制到 64×36 的离屏 Canvas。
2. 读取每个像素的 RGB、亮度和饱和度。
3. 从暗部、全局环境色和高光区得到三组代表色。
4. 在目的地切换过程中插值三组颜色。
5. 同时驱动背景、面板、主文字、次要文字、选中状态和色卡。

总计采样 `64 × 36 × 3 = 6912` 个像素。运行断言会核对图片数量、采样总数和不同图片的校验信息，避免日后退化成写死颜色。

这里的审美关键是**建立颜色角色，而不是直接使用平均色**：

- `shadow` 提供背景深度和稳定性。
- `ambient` 承担场景识别与主要氛围。
- `highlight` 用于可读文字、微光和局部强调。

如果把平均色直接铺满全屏，结果容易灰、脏或刺眼；将采样色分配到不同界面角色，再和少量中性色混合，才能既保留照片气质，又维持信息层级。

### 案例 C：Hover Marquee 的运动必须发生在语义对象内部

第一版改造采用左侧菜单控制右侧独立图片窗口。它在视觉上已经完整，但没有准确表达“hover-activated image marquee menu”：用户看到的是常见的导航加预览面板，而不是菜单项自身变成跑马灯。

最终方案改为三条全宽导航行：

- 默认状态显示编号、目的地名称和方向提示。
- Hover、键盘 Focus 或选中某一行后，该行原内容在原地被重复的图片、名称和坐标轨道取代。
- 三行分别保留独立的 Motion 控制器，轨道方向有轻微差异。
- 状态优先级为 `hover → focus → selected`，鼠标、键盘和持久选择不会互相打架。

这个返工说明了一个重要原则：

> 视觉完成度不能补偿交互语义偏差。效果叫“行内跑马灯”，运动就应该发生在那一行里。

审美并不只是“漂亮”，还包括形式与含义之间的精确对应。越像成熟产品的画面，越容易掩盖语义已经偏离；因此需要在视觉评审时重新对照条目名称和体验句。

## 8. 构图：把 Demo 当成一个产品镜头

### 1. 一张图只保留一个视觉主语

- BlurHash 的主语是“清晰度恢复”。
- Palette 的主语是“照片与环境色共同切换”。
- Marquee 的主语是“当前菜单行变成运动内容”。

标题、编号、元信息、色卡和说明都只能帮助识别主语，不能与主语争夺注意力。判断方法很简单：把画面缩小到目录卡片尺寸，如果第一眼只看到装饰文字，而不是交互变化，层级就错了。

### 2. 用少数稳定规则形成高级感

本次没有依赖复杂装饰，而是反复使用几条稳定规则：

- 大面积安静底色承载一块高信息密度照片。
- 细线和小号等宽信息用于建立编辑感，不使用过多卡片边框。
- 圆角、描边和阴影只保留一种主规格，避免组件各自发挥。
- 主标题与数据之间建立明显字号差，减少中间层级。
- 留白不是平均分配，而是围绕主视觉建立方向性。
- 强调色来自照片或场景，而不是另加一套品牌色。

“高级”通常来自选择更少、关系更清楚，而不是效果更多。

### 3. 运动也需要层级

把所有元素同时动起来，会让 Demo 看起来像动效合集。更稳妥的分层是：

1. **主运动**：条目要证明的机制，例如清晰度恢复、环境色切换、图片轨道移动。
2. **伴随运动**：帮助主运动连续，例如图片交叉淡化、背景颜色插值。
3. **静态锚点**：标题、价格、编号或边界，为观看者提供稳定参照。

每个 Demo 只保留一个主运动。伴随运动的幅度和速度都应弱于主运动。

## 9. Capture 分流：GIF 是三秒叙事，不是录屏附件

Capture 必须根据“核心效果由什么触发”分流，不能为了批量方便把真实交互统一改造成自动播放：

| 类型 | 适用条件 | Capture 输入 | `__setPreviewTime` 的职责 | 不合格做法 |
| --- | --- | --- | --- | --- |
| 自动型 `automatic` | 核心机制无需用户输入，例如持续模拟、加载交接、自动编排、可视化启动 | 固定时间轴、固定种子、固定数据 | 直接设置决定性时间，任意时刻可重现相同帧 | 用真实时间、网络速度或未固定随机数碰运气 |
| 交互型 `interactive` | 名称明确声明 hover、click、focus、drag、scrub、pointer、keyboard、velocity 等输入 | Capture 脚本发送真实 DOM/Pointer/Keyboard 事件 | 只用于在事件发生后稳定推进动画；不能替代触发事件 | 直接给元素加 active class，或让本应 hover 的效果自行轮播 |
| 混合型 `hybrid` | 用户输入启动或改变一段可确定推进的动画，例如点击后时间线、拖拽后惯性 | 先发送真实事件，再按确定时间采样 | 控制事件后的动画进度和结果停留 | 只录自动部分，隐藏用户输入与结果之间的因果链 |

分流判断顺序：

1. 条目名称或 `behavior.trigger` 是否声称用户输入？若是，至少为交互型。
2. 效果的区别是否依赖指针位置、拖拽距离、速度、点击目标或键盘焦点？若是，必须发送真实事件，不能只设置内部状态。
3. 输入之后是否还有可 seek 的动画？若有，采用混合型；若没有，按真实交互后的稳定状态逐帧截图。
4. reduced-motion 会不会改变核心机制？Capture 使用 `no-preference`，但验收仍需单独检查 reduced-motion 路径。

每个 Capture 配置必须固定：入口、320×180 视口、1× DPR、字体与图片等待条件、初始焦点、事件目标、目标框内相对坐标、动作顺序、每段停留、总时长、12 fps、输出路径。拖拽和速度型效果还必须记录起点、终点、持续时间和采样间隔；坐标应从元素 bounding box 计算，不依赖某台机器的屏幕坐标。

Capture 失败时不得静默回退到自动轮播。对于交互型 Demo，GIF 至少要看到“未触发 → 输入发生 → 主体响应 → 可读结果”，而运行页面必须保留用户亲自触发的路径。

三个 GIF 都使用固定的 3 秒录制窗口，共 36 帧。自动型效果使用确定性时间线；交互型效果则由录制脚本执行真实指针动作，使 GIF 展示“移入触发、移出复原”的完整因果链。录制结果为：

| Demo | 帧数 | 不同画面帧 | 文件大小 |
| --- | ---: | ---: | ---: |
| Hover Marquee | 36 | 36 | 约 150 KiB |
| Palette Transition | 36 | 36 | 约 601 KiB |
| BlurHash Reveal | 36 | 15 | 约 260 KiB |

BlurHash 的重复帧来自指针进入前、清晰终态和移出后的停留，这是有意的交互节奏，不代表录制失败。对所有效果机械要求 36/36 不同帧，反而可能迫使本应停留的画面持续无意义运动。

### 一个可靠的短预览节奏

1. **0–20%：建立初始状态。** 观看者要看清“变化前是什么”。
2. **20–70%：发生核心变化。** 不要让标题动画占用最宝贵的时间。
3. **70–90%：呈现结果。** 给眼睛留下理解时间。
4. **90–100%：准备循环。** 通过停留、回落或构图连续性减少跳帧感。

录制前必须等待：

- `document.fonts.ready`
- 所有图片的 `decode()`
- Canvas、算法输出和动画控制器初始化
- Demo 自己的 runtime assertion

录制后要同时检查运行页面和 GIF。GIF 压缩可能改变渐变、细线、暗部和文字边缘；只看浏览器页面并不能代表最终目录卡片的效果。

## 10. 前后证据与双尺度视觉验收

“看起来更好”太主观，不能支持批量协作。更有效的做法是把审美拆成可观察问题。

### 1. 每个 Demo 必须保留前后对比证据

优化前证据必须在第一次写文件前确定；优化后不能再凭记忆描述旧画面。允许两种保存方式：

- **Git 基线方式（默认）**：记录 `<baseline-sha>:<gif-path>`，需要时用 `git show` 恢复，不向仓库重复提交 150 份旧 GIF。
- **研究包方式**：当旧画面不在 Git、需要跨分支评审或基线即将消失时，将 before GIF/PNG 保存到带日期的研究包，并记录来源。

每项对比至少包含：

1. 相同 320×180 尺寸的首个可读状态。
2. 相同叙事位置的核心变化状态，而不是挑选一张对新版有利的任意帧。
3. 稳定结果状态与循环边界。
4. 一段不超过五行的差异说明：修复了什么语义、场景、层级和运动问题。
5. before/after 的帧数、不同画面帧数、文件体积和已知压缩差异。

不得通过不同裁切、不同缩放、额外调色或只给 after 加设备外框来制造提升。若新版只改变主题、配色或素材，而体验句、核心状态和使用价值没有更清楚，前后对比结论必须是“未完成”。

### 2. 缩略图尺度：目录卡片是真正的第一现场

源 GIF 固定为 320×180，但桌面目录中会缩到约 144×81。验收必须在实际目录卡内完成，而不是在图片查看器里把 GIF 放大观察：

- 不看标题，首个可读片段能否辨认主体类型和变化方向。
- 核心变化是否占有足够像素，不依赖 6–8 px 的小字说明。
- 文字、细线、暗部和相近色在 GIF 压缩后是否仍清楚。
- 卡片进入视口、懒加载和循环时是否无白帧、闪屏和跳变。
- 与上下相邻条目并排时，主体行为能否一眼区分，是否像同模板换色。

### 3. 实时 Modal 尺度：可操作性与细节不能被缩略图掩盖

详情页会把实时 Demo 放进最高约 720×405 的 16:9 iframe。验收必须打开目录 Modal，并同时检查桌面、窄屏和至少一次尺寸变化：

- 交互目标位置正确，Hover、Focus、Click、Drag、键盘或触控按契约可用。
- 320×180 的硬编码构图在放大后没有只占左上角、拉伸、裁切错位或过度空白。
- 图片保持合适分辨率与裁切；Canvas/DPR、SVG 和 DOM 边界同步。
- 状态标签和产品信息在大尺度下仍像界面内容，而不是录屏专用贴纸。
- `__PREVIEW_READY__` 之前不显示半成品，资源失败时显示明确错误。
- reduced-motion 与键盘路径在实时页面可用，即使 GIF 使用正常运动偏好。

双尺度验收采用独立结论：`thumbnail: pass|rework` 和 `live_modal: pass|rework`。不得用“缩略图反正看不清细节”放过 Modal 粗糙，也不得用“Modal 很好看”放过目录卡主体不明。

### 4. 通用观察清单

#### 场景与语义

- 不看标题，能否说出这是图片加载、图片取色还是菜单预览？
- 内容是否像真实产品数据，而不是 Lorem Ipsum 式占位？
- 交互是否发生在名称声称的对象上？
- 去掉装饰后，核心机制仍然成立吗？

#### 构图与层级

- 缩小到 320×180 后，主语是否仍然清晰？
- 是否只有一个主要焦点和一个主要运动？
- 文字是否承担信息，而不是充当纹理？
- 是否存在没有功能的徽章、渐变、光晕或边框？

#### 颜色与素材

- 强调色是否来自内容或明确的视觉系统？
- 三个场景既统一又有足够区分吗？
- 图片裁切在所有状态下都保留主体吗？
- 素材是否本地化、可追溯、无文字和 Logo 风险？

#### 动效与交互

- 核心变化是否由真实点击、Hover、Focus、键盘或时间线触发？
- 速度是否允许看清起点、过程和结果？
- 辅助动画是否抢过核心动画？
- 循环是否自然，reduced-motion 和键盘路径是否仍可用？

#### 真实性与工程质量

- 宣称的算法或库是否真的参与了输出？
- 图片、字体和计算未完成时，ready 是否会保持失败？
- 缺少资源时是否明确报错，而不是显示一个看似合理的替代画面？
- Demo、构建产物、GIF 和目录 ID 是否一一对应？

## 11. 多 Agent 共享工作区：并行实现，串行收口

这次采用三个并行工作流：

1. 全量审计 148 个既有 GIF，选择最有审美提升杠杆的候选，并负责 BlurHash。
2. 审查构建与 GIF 捕获链路，并负责 Palette Transition。
3. 审查网页目录与实时预览接线，并负责 Hover Marquee。

并行化适合：候选审计、独立 Demo 实现、捕获链路检查、网页接线检查。以下环节必须集中收口：

- 统一艺术方向和素材规格。
- 检查三个 Demo 是否形成同一套模板。
- 对照条目名称做语义复核。
- 重建同一个 dist、批量录制 GIF、运行全仓库审计。
- 最终决定是否接受或返工。

Marquee 第一版被集中复核否决，就是集中收口的价值。多 Agent 提升速度，但不能让每个局部方案自行定义“完成”；所有方案需要回到同一组体验句和验收门槛。

建议每个并行任务都携带同样的交付契约：

```text
效果名称：
一句话体验：
真实机制：
使用场景：
GIF 中必须出现的三个状态：
键盘 / reduced-motion 路径：
本地素材与来源：
运行断言：
```

上面的三个工作流是本次案例事实；扩展到 100+ Demo 后，多 Agent 必须遵守更严格的共享工作区规则：

1. **先认领，后写入。** 台账中的 `owner` 和 `source_paths` 是文件所有权；同一时刻一个源文件只能有一个实现者。共享 CSS、Capture 脚本、manifest、provenance 和目录数据默认由集成人所有。
2. **实现 Agent 只写已分配源码。** 未经任务明确授权，不运行全量 build、Capture、生成文档，不提交，不 push，不清理其他人的文件。
3. **共享素材由一个素材负责人落盘。** 其他 Agent 可以引用已完成资产，但不得同时覆盖、重压缩或改名。资产尚未稳定时，依赖任务保持 `blocked-on-asset`。
4. **完成消息必须带文件白名单。** 包括实际修改路径、机制说明、ready/runtime assertion、建议 Capture 类型、已做的定向静态检查和未做事项。
5. **收口队列严格串行。** 集成人一次只接管一个 Demo：冻结该 Demo 源码 → 检查 diff → build → Capture → 双尺度 QA → 审计 → 提交 → push → 更新台账，然后才接管下一项。
6. **不得用 Git 命令清理共享工作区。** 发现别人的未跟踪或未提交改动时保留原状；不得 `reset --hard`、`checkout --`、`clean`、随意 stash 或 rebase。路径重叠时停止收口并协调所有者。
7. **全局生成物只有集成人写。** `dist`、GIF、provenance、生成 README 和批次报告会产生跨任务冲突，不能由多个 Agent 并发生成。
8. **收口中发现问题退回原 owner。** 集成人只做小型机械修正；需要重新定场景、重构交互或更换素材时，状态改为 `rework`，解除冻结后再分配。

这套规则的核心不是减少并行，而是把并行限制在边界清楚的源码工作，把所有共享副作用集中到可审计的单通道中。

## 12. 每个 Demo 独立提交与 Push 的 Git 协议

目录级优化必须保持“一个 Demo，一个可回滚提交，一个远端完成点”。即使同一批使用共享素材，也不能把十几个互不相关的效果塞进一个提交。唯一例外是首次加入的一组共享资产：可以在第一个使用它的 Demo 提交中一并进入仓库，后续 Demo 只引用该已提交资产，并在各自提交中保留独立行为变更。

集成人对每个 Demo 按以下顺序执行：

1. 确认上一个 Demo 已 push，台账有提交 SHA；运行 `git status --short`，识别并保留所有非本 Demo 改动。
2. 如果工作区干净且远端可能推进，先 `git fetch` 并确认本地 `main` 可快进；共享工作区有未提交改动时不得擅自 pull、rebase 或 stash。
3. 根据契约列出允许路径，使用显式路径检查 `git diff -- <paths>`；发现范围外修改先追溯所有者。
4. 在当前集成状态下构建，定向 Capture 当前 ID；人工检查源码页面和 GIF，再运行真实性与项目门禁。
5. 使用显式文件路径 `git add`，不得 `git add .`。构建产生的共享 bundle、manifest 或删除项必须逐一确认确由当前 Demo 需要。
6. 检查 `git diff --cached --stat` 和 `git diff --cached`，暂存区只能包含当前 Demo 单元及其必要共享输出。
7. 提交信息采用 `Polish <effect-id> demo` 或同等清晰格式；提交后记录 SHA。
8. 立即 push 到仓库约定分支；验证远端 main 包含该 SHA。push 失败时该项仍是 `qa-pass`，不得标为 `done`，也不得开始下一项收口。
9. 更新台账为 `pushed`，记录 GIF 指标和验收人，再解冻下一个 Demo。

若远端在 push 前出现新提交：

- 工作区干净时，只允许快进同步后重新验证当前 Demo。
- 工作区包含其他 Agent 改动时，停止 Git 操作，先协调；不得为了赶进度搬移或重放不属于当前单元的改动。
- 出现冲突时，当前 Demo 回到 `integration-blocked`。解决冲突后必须重跑 build、Capture 和双尺度验收，旧结果不能沿用。

独立提交不是形式要求。它让 150 项优化可以逐项回滚、二分问题、统计吞吐，也确保一次审美判断失误不会拖着整批一起发布。

## 13. 单 Demo 推荐工作流

### 阶段 1：审计和定基线

1. 浏览同类 Demo 的运行页面和 GIF，而不是只读源码。
2. 找到“真实但表达弱”的条目。
3. 保存优化前 GIF，建立视觉基线。
4. 为每个候选写一句体验描述和三个可见状态。

### 阶段 2：建立艺术方向

1. 决定场景领域，例如旅行、商品、数据工作台或内容阅读。
2. 定义字体、颜色、材质、密度和构图规则。
3. 先准备一小组可复用、本地化、有来源记录的素材。
4. 确认每个 Demo 使用同组素材时仍有不同主体行为。

### 阶段 3：实现真实机制

1. 先让算法、库与真实输入成立。
2. 再用产品场景包装机制，不改变它的语义位置。
3. 加入图片、字体和运行状态的严格 ready 条件。
4. 为关键机制添加 runtime assertion，阻止日后退化成假效果。

### 阶段 4：设计预览时间线

1. 在三秒内安排起点、变化、结果和循环。
2. 保留一个主运动，减少与机制无关的动画。
3. 使用固定时间和确定性输入，避免随机漂移。
4. 在目录缩略图尺寸下预览，而不是只看大窗口。

### 阶段 5：集中验收

1. 浏览器运行页面与生成 GIF 并排检查。
2. 新旧 GIF 并排检查，确认不只是“换主题”。
3. 同批新 GIF 并排检查，确认没有模板化。
4. 对照名称和体验句做最终语义复核。
5. 运行真实性、构建、目录 UI 和文件完整性门禁。

## 14. 批次台账：唯一进度事实源

100+ Demo 不能依靠聊天记录、Agent 记忆或 Git 日志猜测进度。每个批次必须维护一份台账，且只有集成人能把状态改为 `committed` 或 `pushed`。

状态机固定为：

```text
queued → audited → reserved → source-ready → integrating → capture-ready
       → qa-pass → committed → pushed
                    ↘ rework
任意状态 → blocked / dropped
```

- `source-ready`：实现 Agent 已交付源码，但尚未构建和录制。
- `capture-ready`：构建页面、ready、runtime assertion 与 Capture 分流通过。
- `qa-pass`：before/after、缩略图、实时 Modal、GIF 和全局门禁均通过。
- `pushed`：远端 main 已包含独立提交，是唯一等价于“优化完成”的状态。
- `rework`：有明确可修正问题；必须记录退回条件和下一责任人。
- `blocked`：缺素材、依赖、授权或共享文件决策，当前不能安全推进。
- `dropped`：复核后不应优化，例如现状已足够好、风险大于收益或与其他条目重复。

批次总表模板：

| Batch | Effect ID | Priority | Owner | Baseline SHA | Experience | Capture | Asset | Source | Thumb | Modal | Audit | GIF frames/unique/size | Commit | Push | Status | Rework / blocker |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `B01` | `<effect-id>` | `P0/P1/P2` | `<owner>` | `<sha>` | `<一句话>` | `A/I/H` | `reuse/gen/code/official` | `✓/—` | `✓/R/—` | `✓/R/—` | `✓/R/—` | `36/24/420KiB` | `<sha>/—` | `✓/—` | `queued` | `<原因>` |

每个批次还要记录：

- 批次目标和停止条件。
- 艺术方向、共享素材清单及负责人。
- Capture 类型数量，防止整批只覆盖容易录制的自动型效果。
- 开始/完成/返工/丢弃数量，以及每项独立提交 SHA。
- 完全重复和感知相似检查结果。
- 批次结束时 150 项全局真实性与项目校验结果。

台账更新时机：认领、源码交付、进入集成、视觉退回、验证通过、提交、push。不得在批次结束时一次性补写，因为那会丢失失败与返工轨迹。

## 15. 100+ Demo 推进策略

### 1. 先做全量盘点，不按文件顺序盲推

对 150 项建立只读清单，并至少记录：机制真实性风险、语义表达缺口、320×180 识别度、实时交互质量、素材需求、Capture 类型、同类相似度、预计改造成本。按以下优先级排序：

- **P0 修复型**：机制或素材与条目声明不一致、真实交互被自动状态冒充、GIF 与源码错配。先修复或下架，不能当作普通美化排队。
- **P1 高杠杆型**：机制真实，但抽象占位、弱场景或错误构图让使用价值不可见；本次三个案例属于这一层。
- **P2 打磨型**：语义和场景已经成立，只需要解决缩略图层级、动效节奏、响应式或压缩问题。
- **P3 保持型**：现状已清楚且有辨识度。记录“无需优化”也是有效审计结果，不为覆盖率强行改版。

### 2. 用代表性小样校准门槛

在扩批前先完成 3–5 个试点，至少包含：一个自动型、一个交互型、一个混合型；一个图片输入、一个代码原生视觉；一个 DOM/SVG 和一个 Canvas/WebGL 表面。试点全部走到独立 push，确认契约、Capture、双尺度 QA、台账和 Git 协议可行后再扩大。

任何共同基础设施改动先在试点验证。Capture 脚本、共享 CSS 或 manifest 仍频繁变化时，不得同时铺开几十个 Demo，否则后续会产生大面积重录和视觉漂移。

### 3. 以 8–12 项为一波，而不是一次打开 150 项

每波建议 8–12 个 Demo，按机制和素材依赖分组，但混合不同 Capture 类型与视觉类别。单波流程：

1. 全量盘点选出候选并冻结体验句。
2. 素材负责人先完成共享素材；ImageGen 资产先过缩略图审查。
3. 2–3 个实现 Agent 并行处理互不重叠的源码，集成人保持空闲以接收完成项。
4. 集成人按队列一次收口一个 Demo，并在每项后独立 commit/push。
5. 每完成 3 项做一次相邻缩略图联系表，防止风格和模板逐渐趋同。
6. 每波结束运行 150 项全量审计和目录 UI 测试，更新台账与下一波优先级。

建议 WIP 上限：每个实现 Agent 同时只有一个 `reserved` Demo；集成队列不超过三个 `source-ready` 项；任一项进入 `rework` 后优先关闭再开新项。优化吞吐以 `pushed Demo / 周期` 统计，不以“写过源码”或“生成过 GIF”统计。

### 4. 用视觉簇保持方向，用机制差异阻止模板化

可以建立旅行、商品、编辑阅读、创意工具、数据工作台、声音与空间等视觉簇，共享少量色彩、字体或素材方向。每个簇仍必须满足：

- 相邻 Demo 的主语、构图骨架和运动轨迹不能实质相同。
- 同一素材在不同 Demo 中承担不同机制角色，而不是只换标题。
- 每完成三项重新查看 320×180 联系表；若不看标题难以区分，暂停该簇并重做至少一项。
- 不为“全站统一”抹掉需要不同密度、速度和界面语言的机制差异。

### 5. 用数据监控质量，而不是用数量掩盖返工

每波至少统计：`pushed` 数、一次通过率、平均返工轮数、自动/交互/混合 Capture 比例、ImageGen 使用率、GIF 体积分布、缩略图失败数、Modal 失败数、真实性审计失败数和相似预览告警数。若同一失败原因在一波内出现两次，暂停新增并修正规范、脚本或共享组件，再继续扩批。

150 项完成的定义不是台账中 150 个 `pushed`；还要有 150 / 150 的真实性与项目校验、目录双尺度抽检、无未解释的重复预览，以及所有提交均可按 Effect ID 独立定位。

## 16. 失败、阻断与返工条件

以下情况是硬失败，必须停止当前 Demo 的提交或 push：

| 条件 | 状态 | 必须动作 |
| --- | --- | --- |
| 声称的库、算法或 API 没有参与主体输出 | `blocked` | 重新实现真实机制；不能改标题迁就假实现 |
| 交互对象、触发层或结果与名称/体验句不一致 | `rework` | 回到场景与信息架构，不做表面润色 |
| 图片、字体、模型或计算未完成时 ready 已成立 | `rework` | 收紧等待条件并补失败路径 |
| 交互型效果用内部状态或自动轮播冒充真实输入 | `rework` | 改 Capture 路径并重录全部帧 |
| GIF 与运行 Demo 的主体、时序或构图不一致 | `blocked` | 从已验证构建页面重新 Capture |
| before 基线不可恢复，或前后对比尺度不一致 | `blocked` | 补齐可信证据后再评审 |
| 320×180 主体不可辨、首段空白、核心动作太晚 | `rework` | 重排构图与三秒时间线 |
| 实时 Modal 交互失效、放大错位、Canvas 拉伸或资源报错 | `rework` | 修响应式与运行路径，不能只保留好看的 GIF |
| ImageGen 素材含伪文字、Logo、水印、明显结构错误或被描述为真实事实 | `rework` | 修图、重生成、换授权素材或缩小场景 |
| 与其他预览完全重复或主体行为高度相似且无充分解释 | `blocked` | 重定场景/构图，或取消收录 |
| 构建、真实性审计、项目校验、目录 UI 或 `git diff --check` 失败 | `blocked` | 修复并重跑相关门禁及受影响全局检查 |
| 暂存区混入其他 Demo 或其他 Agent 改动 | `integration-blocked` | 取消当前收口，按所有权拆分；不得提交混合范围 |
| 独立提交未 push 或远端不包含 SHA | `qa-pass` | 解决 Git/远端问题；不得宣称完成或接续下一收口 |

以下视觉信号虽然未必触发脚本失败，也必须返工：

- 必须读标题或小字才知道画面在做什么。
- 装饰动画比核心机制更显眼，或多个元素争夺主运动。
- 新版只是换图、换色、加圆角，体验问题仍未被说明。
- 结果状态没有停留时间，循环边界造成明显跳帧。
- 缩略图通过但 Modal 像放大的录屏布景，或 Modal 完整但缩略图只剩文字纹理。
- 同一批作品逐渐变成相同的左标题、右卡片、顶部胶囊模板。
- GIF 为减小体积而破坏渐变、细线、暗部或文字；默认以通过仓库门禁并尽量小于 1 MiB 为工作目标，但不能用失真换体积。

返工必须在台账写清“失败门、可观察现象、责任人、重新验收范围”。只要改动影响了输入、时序、构图或素材，旧 GIF、旧双尺度结论和旧 runtime assertion 结果都作废；不能只重跑最后一个命令。

## 18. 可以复用的审美优化原则

1. **先修语义，再修样式。** 交互发生的位置比阴影参数更重要。
2. **先找场景，再找素材。** 场景决定图片需要表达什么，而不是反过来。
3. **让内容参与算法。** 图片不仅要被展示，还可以成为 BlurHash、取色、裁切或导航反馈的真实输入。
4. **一个 Demo 只有一个主语。** 所有静态信息和伴随动画都为它服务。
5. **统一不等于复制。** 可以共享艺术方向和素材，但主体行为必须一眼不同。
6. **漂亮不能替代准确。** 如果视觉方案偏离条目名称，应当返工，而不是改标题解释。
7. **把 GIF 当成最终产品。** 页面、捕获时序、压缩和缩略图识别度要一起设计。
8. **让主观判断留下证据。** 体验句、前后对比、运行断言、采样数量和视觉验收记录都能让审美协作更稳定。

最终目标不是让每个 Demo 都像一张设计作品，而是让观看者在很短时间内同时感到两件事：**这个效果是真的，以及我知道它可以用在哪里。**
